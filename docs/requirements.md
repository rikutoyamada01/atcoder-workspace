# Competitive Programming Workspace - 要件定義書 (MVP版)

## 1. はじめに (Introduction)

### 1.1 目的
本要件定義書は、AtCoderに初めて触れる初心者が、ローカルでの環境構築を一切行わずに、ブラウザの単一タブ内で「問題閲覧 → コーディング → テスト → 提出」までをストレスフリーに行えるChrome拡張機能「Competitive Programming Workspace（以下、本システム）」のMVP（最小限の実用製品）としての機能および非機能要件を定義します。

**本システムにおけるMVPの成立条件は、「一画面（単一のタブ）で問題文、エディタ、テスト、および提出のすべての操作が完結すること」です。** このコア体験の提供を最優先とするため、自動提出までのフローをPhase 1（MVP）のスコープとし、テンプレート＆スニペット機能等はフェーズ2として定義します。

### 1.2 背景とコア価値
初心者が競プロを始める際の最大の障壁である「C++等のコンパイル実行環境の構築」「エディタとブラウザの往復ストレス」「サンプルケースの手動テスト」を解決します。
本拡張機能は、**「インストールするだけで即座にAtCoderが自分の開発環境になる」**というゼロセットアップ体験を提供します。

## 3. MVP 機能要件 (MVP Functional Requirements)

### F-1: AtCoder問題ページへのエディタ埋め込みとナビゲーション (Split View & Navigation)
* **対象ページ**: AtCoderの問題ページ (`/contests/*/tasks/*`) にサイドパネルを埋め込みます。
* **左右2分割表示**:
  * 左側に公式の問題文（AtCoderの既存DOM）、右側に本システムのエディタパネルをスプリット表示。
  * スプリットバーをドラッグすることで比率を変更可能。
  * サイドパネルは「開閉ボタン（トグル）」で瞬時に折りたたみ可能。
* **問題のクイックナビゲーション**:
  * エディタ上部または下部に「前の問題へ戻る」「次の問題へ進む」ボタンを設置。
  * 例：`/tasks/abc300_a` を開いている際、「次の問題へ進む」を押すと、問題一覧画面を経由せずに直接 `/tasks/abc300_b` へ遷移。
* **画面の永続化**: ユーザーが設定した左右の分割比率はブラウザに保存され、別問題を開いた際も維持されます。

### F-2: 埋め込みコードエディタ (Monaco Editor)
* **コアエディタ**: ブラウザ上でVS Codeと同等の操作感を提供する `Monaco Editor` を採用。
* **初心者向け制限（重要）**:
  * 自力でのコーディング力を養うため、**入力補完（IntelliSense）およびAI支援機能は一切なし**とします。
* **サポートされる編集機能**:
  * C++ / Python / Rust / Java などの主要言語に対応したシンタックスハイライト。
  * 自動インデント。
  * 括弧の自動補完。
  * コードの自動整形機能。
* **言語切り替え**:
  * エディタ上部にドロップダウンメニューを設置し、プログラム言語を選択可能（選択状態はAtCoder公式の言語選択と連動）。

### F-3: コードのローカル自動保存 (Auto Save)
* **自動保存**: エディタで記述中のコードは、キー入力が一定時間（1.5秒）停止したタイミング、またはタブが閉じられる/切り替わる瞬間に自動保存されます。
* **ストレージ保存**: Chromeの `chrome.storage.local` を使用してデータを保存。
* **自動復元**: 再び同じ問題ページを開いた際、最後に編集していたコードが自動的にエディタへ復元されます。

### F-4: ワンクリック自動テスト (Auto Test Runner)
* **サンプルケースの自動抽出**:
  * 問題ページ内のHTMLから「入力例 1」「出力例 1」などのテキストデータを自動で抽出。
* **コードの実行とテスト判定**:
  * ユーザーが「テスト実行」ボタンを押すと、裏でAtCoder公式の「カスタムテスト (Custom Test)」機能のAPI（`/contests/<contest_id>/custom_test/submit/json` 等）を呼び出し、現在のエディタ内のコードと抽出したサンプル入力データを送信して実行。
  * 実行完了後、返ってきた出力結果と公式の「出力例」を自動比較し、テストケースごとに `AC (一致)` または `WA (不一致/エラー)` を表示。

### F-5: 自動提出とポップアップ表示 (Auto Submitter & Popup)
* **シームレスな提出**:
  * エディタ下部に「提出 (Submit)」ボタンを配置。
  * ボタンが押されると、現在のコードと選択された言語情報をAtCoder公式の提出用URLに対し、バックグラウンドで送信。
* **提出完了ポップアップ**:
  * 提出が正常に受け付けられると、エディタ画面内にインラインポップアップを表示します。
  * ポップアップ内には、提出結果を確認できる「提出結果ページ（`/submissions/me`）へのリンクボタン」を設置。
  * **タブ移動の制限**: 提出ボタンを押したあとに強制的なタブ移動は発生させず、ユーザーが自分の意志でポップアップ内のリンクをクリックした場合のみ移動する設計とします。

### F-6: 実行・提出結果のセッション保持 (Console State Persistence)
* **コンソール状態の自動復元**:
  * ユーザーが前後の問題へナビゲーションボタンで遷移したり、別ページを表示したりして元の問題に戻ってきた際、ブラウザの `sessionStorage` に一時保存した直近のテスト実行結果やジャッジの提出結果ログを自動でコンソールに復元・再レンダリングします。これにより、別タブや問題の切り替えによるテスト結果の消失を防ぎます。

### F-7: 解答ステータスの個別管理 (Problem Status Management)
* **自力 / 解説ACの記録**:
  * 解答がACした際、それが自力で解けたもの（`self_ac`）なのか解説を読んで解いたもの（`editorial_ac`）なのかを記録します。
  * 既にAC済み（解説AC等）である問題に再度提出してACした場合に、自動的にステータスが `self_ac` 等に書き換わって上書きされないよう、現在のステータスが未解決 (`unsolved` または未登録) の場合のみ自動更新を行うガード処理を設けます。
* **オプション画面での管理と復習支援**:
  * 拡張機能オプション画面に、これまでACした問題のステータス管理テーブルを表示します。
  * テーブル内の「コンテスト名」「問題名」はAtCoderの対応ページへの直接リンク（通常は親テキスト色と同じで、マウスホバー時のみ下線が引かれる視覚的にシンプルなデザイン）として機能させ、手軽に復習へ移れるようにします。
  * 「すべて / 自力AC / 解説AC」によるステータス絞り込みドロップダウンを追加し、特定の解答状態の問題だけを抽出表示できるようにします。

## 4. 将来機能要件 (Future Roadmap Requirements)

### 【フェーズ 2】学習の定着とエラー分析機能（後日実装）
* **全提出履歴の自動記録**:
  * AC（正解）だけでなく、WA（誤答）、TLE（実行時間制限超過）、MLE（メモリ制限超過）、RE（実行時エラー）、CE（コンパイルエラー）など、すべての提出結果（ジャッジステータス）を自動的に検知して保存。
* **エラー傾向分析機能**:
  * 蓄積された履歴データから、「どの種類のエラーを多く作ってしまっているか」を可視化・分析。
* **学習メモ機能 (Note)**: エディタと切り替える形で「学習メモ」を入力できるタブを設置。マークダウン形式で「つまずいた点」をその場に書き残せる。
* **学習ダッシュボード**: 総AC数、週別のAC数、毎日継続して解いているかを示すカレンダー（草生やし機能）を提供。

### 【フェーズ 3】中級者へのステップアップ機能（後日実装）
* **解法ライブラリ**: 過去に解いた問題のコードやメモを、「二分探索」「DP」などのタグやキーワードで高速検索できる機能。
* **復習キュー (Review Queue)**: 過去に間違えた問題や、復習したい問題を自動的にストックし、適切なタイミングで提示する機能。
* **Markdown一括エクスポート**: 蓄積したコードやメモを、Obsidianなどの個人ノートツールにそのままインポートできる形式（YAMLフロントマター付きMarkdown）でZIPダウンロードできる機能。

## 5. 非機能要件 (Non-Functional Requirements)

### N-1: ゼロインストール・ゼロコンフィグ (Zero Setup)
* Chromeウェブストアから拡張機能を追加するだけで、すべての機能が使えること。別途外部アプリケーションやランタイム（Node.js, C++コンパイラ等）のインストールを一切求めない。

### N-2: パフォーマンスの重視
* AtCoderの初期表示速度を落とさないため、Monacoエディタの各種ライブラリは、ユーザーが最初にサイドパネルを開く（または初回呼び出し）まで非同期で遅延ロード（Lazy Load）する。

### N-3: ユーザビリティと表示最適化
* 初心者のPC画面（横幅が狭いノートPCなど）でも問題文とエディタが重ならずに閲覧できるよう、フォントサイズ変更機能やスプリット表示の折りたたみを容易にする。
* AtCoderのダークテーマ等の表示に自動で適応する（Monaco Editorもダークモードになる）。

## 6. MVP 推奨データモデル (Schema Design)

MVP段階では、複雑なリレーショナルDBは不要とし、`chrome.storage.local` にキー・バリュー形式でシンプルにデータを保存します。

### キー構造
* **コード保存**:
  `code:{contest_id}:{problem_id}:{language}` -> `String` (記述されたソースコード)
* **直近の選択言語**:
  `settings:last_selected_language` -> `String` (最後に使用した言語ID)
* **スプリット幅比率**:
  `settings:split_ratio` -> `Number` (左右の表示割合、例: 0.5)
